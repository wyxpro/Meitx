# 文档与架构决策记录（ADR）

## 概述

记录决策，而不只是代码。最有价值的文档捕捉的是*为什么*——导致某个决策的背景、约束和权衡。代码展示*构建了什么*；文档解释*为什么这样构建*，以及*考虑过哪些替代方案*。这些上下文对未来在代码库中工作的人类和 AI 助手都至关重要。

## 适用场景

- 做出重大架构决策
- 在竞争方案之间做选择
- 新增或变更公共 API
- 发布改变用户可见行为的功能
- 新成员（或 AI 助手）入职项目
- 发现自己在反复解释同一件事
- 撰写文档、提案、技术规范或决策记录

**不适用场景：** 不要记录显而易见的代码。不要添加只是重述代码的注释。不要为一次性原型写文档。

## 架构决策记录（ADR）

ADR 记录重大技术决策背后的推理过程，是你能写的最有价值的文档。

### 何时写 ADR

- 选择框架、库或重大依赖
- 设计数据模型或数据库 schema
- 选择认证策略
- 决定 API 架构（REST vs. GraphQL vs. tRPC）
- 选择构建工具、托管平台或基础设施
- 任何逆转代价高昂的决策

### ADR 模板

将 ADR 存放在 `docs/decisions/` 目录，按序号命名：

```markdown
# ADR-001: 使用 PostgreSQL 作为主数据库

## 状态
已接受 | 被 ADR-XXX 取代 | 已废弃

## 日期
2025-01-15

## 背景
我们需要为任务管理应用选择主数据库。核心需求：
- 关系型数据模型（用户、任务、团队及关联关系）
- 任务状态变更需要 ACID 事务
- 支持对任务内容的全文搜索
- 有托管服务可用（小团队，运维资源有限）

## 决策
使用 PostgreSQL + Prisma ORM。

## 考虑过的替代方案

### MongoDB
- 优点：schema 灵活，上手容易
- 缺点：我们的数据天然是关系型的；需要手动管理关系
- 拒绝原因：在文档数据库中存关系型数据会导致复杂的 join 或数据冗余

### SQLite
- 优点：零配置，嵌入式，读取快
- 缺点：并发写入能力有限，生产环境无托管服务
- 拒绝原因：不适合多用户 Web 应用生产环境

### MySQL
- 优点：成熟，支持广泛
- 缺点：PostgreSQL 的 JSON 支持、全文搜索和生态工具更好
- 拒绝原因：PostgreSQL 更符合我们的功能需求

## 影响
- Prisma 提供类型安全的数据库访问和迁移管理
- 可以使用 PostgreSQL 全文搜索，无需引入 Elasticsearch
- 团队需要 PostgreSQL 知识（通用技能，风险低）
- 托管在 Supabase、Neon 或 RDS
```

### ADR 生命周期

```
已提出（PROPOSED） → 已接受（ACCEPTED） → 已取代（SUPERSEDED）或已废弃（DEPRECATED）
```

- **不要删除旧 ADR。** 它们保存了历史上下文。
- 决策变化时，写一个新 ADR，引用并取代旧的。

## 文档协作工作流

指导用户完成协作文档创建的结构化工作流，分三个阶段：上下文收集、细化与结构、读者测试。

### 何时启用此工作流

**触发条件：**
- 用户提到撰写文档："写一个文档"、"起草提案"、"创建规范"、"写一下"
- 用户提到具体文档类型："PRD"、"设计文档"、"决策记录"、"RFC"
- 用户似乎在开始一项大型写作任务

**初始提示：**
向用户提供协作文档创建的结构化工作流，说明三个阶段：
1. **上下文收集**：用户提供所有相关背景，同时回答澄清性问题
2. **细化与结构**：通过头脑风暴和编辑逐节迭代构建内容
3. **读者测试**：用全新 AI 实例（无上下文）测试文档，在他人阅读前发现盲点

如果用户拒绝，自由发挥。如果接受，进入第一阶段。

### 第一阶段：上下文收集

**目标：** 缩小用户已知和 AI 已知之间的差距，为后续提供明智的指导。

#### 初始问题

首先询问文档的元信息：
1. 这是什么类型的文档？（如技术规范、决策记录、提案）
2. 主要读者是谁？
3. 读者读完后希望达到什么效果？
4. 是否有模板或特定格式要求？
5. 还有什么约束或背景需要了解？

告知他们可以用简短的方式回答，或以最适合自己的方式倾倒信息。

**如果用户提供了模板或提到了文档类型：**
- 询问是否有模板文档可分享；如果提供了链接或文件，读取它

**如果用户提到编辑现有共享文档：**
- 读取当前状态；检查是否有没有 alt 文本的图片；如有，询问是否需要生成 alt 文本

#### 信息倾倒

鼓励用户倾倒所有上下文，包括：项目/问题背景、相关团队讨论、为何不用其他替代方案、组织上下文、时间约束、技术架构、利益相关方顾虑。

告知不必担心整理——先全部输出。

#### 澄清性问题

用户完成初始信息输出后，根据上下文空白生成 5-10 个编号问题。

**退出条件：** 当能询问边缘情况和权衡时（无需解释基础知识），上下文已足够。

询问是否还有更多上下文，或准备进入起草阶段。

### 第二阶段：细化与结构

**目标：** 通过头脑风暴、筛选和迭代优化，逐节构建文档。

从未知因素最多的节开始。确认结构后，创建带占位文本（[待撰写]）的 Markdown 文件。

**每节六步：**

1. **澄清性问题** — 提出 5-10 个关于该节应包含什么的问题
2. **头脑风暴** — 生成 5-20 个编号选项
3. **筛选** — 用户指出保留/删除/合并（"保留 1,4,7"/"删除 3，与 1 重复"）
4. **空白检查** — 询问是否有重要内容遗漏
5. **起草** — 用实际内容替换占位文本
6. **迭代优化** — 用字符串替换修改（绝不重新打印整个文档）；连续 3 次无实质变化后，询问是否有内容可删

**接近完成（80% 以上章节）时：** 重读整个文档，检查流畅性、冗余、矛盾和套话。

### 第三阶段：读者测试

**目标：** 验证文档对真实读者有效，发现盲点。

**步骤 1：** 生成 5-10 个读者会实际提问的问题
**步骤 2（有子 Agent 时）：** 用全新 AI 实例逐一测试，汇总对错
**步骤 2（无子 Agent 时）：** 提供手动测试说明，让用户在新 AI 对话中粘贴文档并提问
**步骤 3：** 额外检查歧义、错误假设、矛盾
**步骤 4：** 报告问题并返回细化阶段修复

**退出条件：** 读者 AI 能持续正确回答问题且不再出现新空白。

**完成后建议：**
- 自己做最终通读，核查事实和链接
- 考虑在附录中链接本次对话
- 根据真实读者反馈持续更新文档

## 内联文档

### 何时写注释

注释*为什么*，而非*是什么*：

```typescript
// 错误：重述代码
// 将计数器加 1
counter += 1;

// 正确：解释非显而易见的意图
// 限流使用滑动窗口——在窗口边界重置计数器，
// 而非按固定计划，以防止在窗口边缘的突发攻击
if (now - windowStart > WINDOW_SIZE_MS) {
  counter = 0;
  windowStart = now;
}
```

### 何时不写注释

```typescript
// 不要注释自解释的代码
function calculateTotal(items: CartItem[]): number {
  return items.reduce((sum, item) => sum + item.price * item.quantity, 0);
}

// 不要留下应该现在就做的 TODO 注释
// TODO: 添加错误处理 ← 直接加上去

// 不要留下注释掉的代码
// const oldImplementation = () => { ... } ← 删掉它，git 有历史记录
```

### 记录已知坑点

```typescript
/**
 * 重要：此函数必须在首次渲染前调用。
 * 如果在 hydration 后调用，会导致无样式内容闪烁，
 * 因为主题上下文在 SSR 期间不可用。
 *
 * 完整设计原理见 ADR-003。
 */
export function initializeTheme(theme: Theme): void {
  // ...
}
```

```typescript
 */

  // ...

```

### OpenAPI / Swagger（REST API）

```yaml
paths:
  /api/tasks:
    post:
      summary: 创建任务
      requestBody:
        required: true

          description: 任务已创建
        '422':
          description: 校验错误
```

## README 结构

每个项目都应有一个涵盖以下内容的 README：

```markdown
# 项目名称

一段话描述项目是什么、做什么。

## 快速开始

1. 克隆仓库
2. 安装依赖：`npm install`
3. 配置环境变量：`cp .env.example .env`
4. 启动开发服务器：`npm run dev`

## 命令

| 命令 | 描述 |
|------|------|
| `npm run dev` | 启动开发服务器 |
| `npm test` | 运行测试 |
| `npm run build` | 生产构建 |
| `npm run lint` | 运行代码检查 |

## 架构

项目结构和关键设计决策的简要概述。
详细内容链接到 ADR。

## 贡献

如何贡献、编码规范、PR 流程。
```

## 面向 AI 助手的文档

AI 助手上下文的特别说明：
- **CLAUDE.md / 规则文件** — 记录项目约定，让 AI 助手遵循
- **Spec 文件** — 保持规范更新，让 AI 助手构建正确的东西
- **ADR** — 帮助 AI 助手理解过去决策的原因（防止重复决策）
- **内联坑点说明** — 防止 AI 助手陷入已知陷阱

## 常见借口与现实

| 借口 | 现实 |
|------|------|
| "代码是自文档化的" | 代码展示是什么，不展示为什么、拒绝了哪些替代方案或适用哪些约束。 |
| "等 API 稳定了再写文档" | 写文档会让 API 更快稳定。文档是设计的第一次测试。 |
| "没人读文档" | AI 助手会读。未来的工程师会读。三个月后的你会读。 |
| "写 ADR 是额外开销" | 10 分钟的 ADR 能防止六个月后关于同一决策的 2 小时争论。 |
| "注释会过时" | *为什么*的注释是稳定的。*是什么*的注释会过时——这就是为什么只写前者。 |

---

## 红色警报

- 架构决策没有书面理由
- 公共 API 没有文档或类型
- README 没有说明如何运行项目
- 注释掉的代码而非删除
- 已存在数周的 TODO 注释
- 有重大架构选择的项目中没有 ADR
- 文档重述代码而非解释意图

## 上线前验证

文档完成后：
- [ ] 所有重大架构决策都有 ADR
- [ ] README 涵盖快速开始、命令和架构概述
- [ ] API 函数有参数和返回值类型文档
- [ ] 已知坑点在相关位置内联记录
- [ ] 没有残留的注释代码
- [ ] 规则文件（CLAUDE.md 等）是最新且准确的
- [ ] 文档已用全新读者测试（第三阶段读者测试）

## 有效指导的建议

**语气：**
- 直接、程序化
- 简要解释影响用户行为的原理
- 不要"推销"这种方式——直接执行

**处理偏差：**
- 用户想跳过某阶段时：询问是否要跳过并自由发挥
- 用户似乎沮丧时：承认这比预期花了更长时间，建议加快方式
- 始终给用户调整流程的主动权

**上下文管理：**
- 全程主动询问提到但缺失的上下文
- 不要让空白积累——遇到时立即处理

**质量优先于速度：**
- 不要急于推进各阶段
- 每次迭代都应有实质性改进
- 目标是一个对读者真正有效的文档
