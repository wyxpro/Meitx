# 移动端功能设计隐私合规增强

## 概述

在生成 App 或微信小程序时，根据功能特征自动推断并注入隐私合规功能模块，输出结构化合规实现指引。本 skill 自动检测目标平台，分别适用 App 或微信小程序的合规规则。

**适用范围：** 功能设计层面的隐私合规（UI 交互流程、权限/授权申请时机、信息收集告知、用户操作路径等），不覆盖隐私政策文本撰写、数据安全架构等其他维度。

## Pre-Step — 平台检测

收到用户需求后，首先判断目标平台：

**小程序关键词：** 小程序、微信小程序、mini program、wx.authorize、scope授权、getPhoneNumber、公众号、wxml、wxss、做个小程序、创建小程序、开发小程序、做小程序

**App 关键词：** App、iOS、Android、原生应用、APK、应用商店、上架、Native、移动应用、手机App、做个App、App开发、应用开发、创建应用、开发应用、做应用

**路由规则：**
- 命中小程序关键词 且 未命中 App 关键词 → **小程序模式**（M 系列组件，引用 `references/miniprogram-components.md`）
- 命中 App 关键词 且 未命中小程序关键词 → **App 模式**（C 系列组件，引用 `references/app-components.md`）
- 都未命中 或 同时命中 → **主动询问用户**：

> 检测到您的需求中未明确指定平台类型。请确认目标平台：
> - **App**（原生移动应用，通过应用商店安装）
> - **微信小程序**（运行在微信内，通过微信打开）

确定平台后进入以下 4 步工作流。

## Step 2 — 生成 App / 小程序

### App 模式

按用户需求生成业务页面后，将合规组件按以下顺序注入：

```
1. C01 隐私政策同意弹窗 → 启动流程第一步
2. C06 权限申请弹窗 → 对应功能首次触发时（仅当涉及系统权限时注入）
3. C05 应用权限说明查看入口 → 设置页面（仅当 C06 触发时注入）
4. C07 个性化推荐关闭入口 → 设置/隐私设置页（仅当含推荐/信息流时注入）
5. C02 隐私政策查看入口 → 设置页面
6. C03 双清单 → 设置页面 + 详情页
7. C04 账号注销流程 → 设置 > 账号管理 > 账号注销（仅当触发注册登录时注入）
8. C09 未成年人保护 → 注册流程或设置页（仅当面向未成年人时注入）
9. C10 自动续费服务告知 → 开通页面（仅当含自动续费/付费时注入）
10. C11 应用分发平台规范 → 下载页面（仅当含应用分发功能时注入）
```

生成每个合规组件时，须遵循 `references/app-components.md` 中定义的设计规范。

### 小程序模式

按用户需求生成业务页面后，将合规组件按以下顺序注入：

```
1. M01 隐私政策弹窗 → 首次进入、收集信息之前
2. M03 授权弹窗 → 用户实际触发对应功能时（不可启动即弹）
3. M04 敏感信息单独告知 → 收集敏感信息的页面
4. M05 首页登录规范 → 登录页/首页（仅当含登录/注册时注入）
5. M06 个性化推荐关闭入口 → 设置/隐私设置页（仅当含推荐/信息流时注入）
6. M02 隐私政策查看入口 → 设置/个人中心（路径≤4层）
7. M07 双清单查看入口 → 设置/个人中心
8. M08 未成年人保护 → M01 之后、进入业务功能之前（仅当面向未成年人时注入）
9. M09 账号注销流程 → 设置 > 账号管理（仅当含注册/登录时注入）
10. M10 自动续费服务告知 → 会员开通页（仅当含自动续费/付费时注入）
```

**特别规则（详见 M05 首页登录规范）：** 面向公众的小程序不得首页打开即要求登录，应先展示内容，仅在用户触发需登录功能时才要求授权；仅供特定人员使用的小程序可首页登录但须明示使用范围。

**M03 授权弹窗实现要求（微信原生/官方弹窗优先原则）：**

M03 实际涉及**两类并存的授权**，须区分处理，且均**优先复用微信能力**：

**A. 系统能力授权（位置、录音、摄像头等）：**
- 在功能触发时调起微信原生授权弹窗，弹窗由微信系统自动渲染（含小程序名称、授权名称、用途说明、等权重的「拒绝」/「允许」按钮）
- **不需要为此类授权编写任何弹窗界面**，且不得用自定义弹窗替代微信原生授权
- 用途说明须在微信侧为每个授权项配置清晰话术
- 降级处理：仅在用户拒绝后，可用自定义界面引导用户前往设置开启，或提供手动输入等替代方案

**B. 用户隐私保护授权（处理个人信息的合规框架）：**
- **优先采用微信官方隐私弹窗**：启用隐私功能后微信自动向用户展示，无需开发者自行设计弹窗即满足合规，这是本 skill 的默认推荐做法
- 仅当需自行控制弹窗样式/时机时，才采用自定义隐私弹窗，且其「同意」按钮必须使用微信指定的隐私授权按钮（详见 `references/miniprogram-components.md` 的 M03-B）

生成每个合规组件时，须遵循 `references/miniprogram-components.md` 中定义的设计规范。

### 合规实现指引（两种模式通用）

每个注入的合规组件须附带结构化实现指引（内部 metadata），包含：组件编号、组件名称、法规依据、触发条件、交互要求、状态流转、开发注意事项。

## Step 3 — 反面模式自检

### App 模式

- [ ] 同意/不同意按钮是否等大等高？
- [ ] 是否有预勾选的复选框？（应默认未勾选）
- [ ] 是否存在“继续/点击即同意”等默示同意（无明确的同意/不同意选择）？
- [ ] 权限申请是否仅在功能触发时出现？（非启动时一次性）
- [ ] 权限拒绝后基础功能是否仍可用？
- [ ] 设置页是否包含所有必备隐私入口？
- [ ] 注销入口是否在 5 次操作内可找到？
- [ ] 自动续费是否有独立勾选框且默认未勾选？（禁止“点击即同意”的默示同意）
- [ ] 登录页是否有协议勾选提示且默认未勾选？

### 小程序模式

- [ ] 隐私授权弹窗是否提供明确的同意/拒绝选项？
- [ ] 勾选框是否默认未勾选？
- [ ] 是否存在“继续/点击即同意”等默示同意（无明确的同意/不同意选择）？
- [ ] 授权是否仅在实际触发功能时申请？（非启动时一次性）
- [ ] 拒绝授权后无关功能是否仍可用？
- [ ] 面向公众的小程序是否避免首页即要求登录？
- [ ] 隐私政策路径是否≤4层？
- [ ] 是否提供双清单查看入口？
- [ ] 注销入口是否在 5 次操作内可找到？
- [ ] 注销/取消按钮是否等视觉权重？
- [ ] 系统能力授权是否通过微信原生授权弹窗实现？（不得自定义弹窗替代微信原生授权）
- [ ] 处理个人信息是否优先采用微信官方隐私弹窗？
- [ ] 自动续费是否有独立勾选框且默认未勾选？（禁止“点击即同意”的默示同意）

## 快速决策参考

- **不确定是否需要？→ 包含它。** 缺一个可能导致下架或封禁。
- **不确定按钮视觉权重？→ 让它们一样大。**
- **不确定权限/授权何时申请？→ 在功能触发时申请。** 永远不要启动时一次性索取。
- **小程序能用微信能力实现？→ 优先复用微信接口。**
- **仅客户端能实现？→ 不回传个人信息至后台。**
